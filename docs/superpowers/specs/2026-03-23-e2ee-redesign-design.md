# E2EE Key Exchange Redesign

## Problem

The current E2EE implementation has critical security gaps that prevent it from being truly zero-knowledge for the gateway:

1. **Gateway sees content public key** — The bridge registers its content public key with the gateway, and the browser fetches it from the gateway. A compromised gateway can substitute its own key (MITM attack on the DEK exchange).
2. **Encrypted path is a stub** — `e2ee_bridge.rs:297-299` returns early on encrypted payloads. The frontend already has DEK encrypt/decrypt plumbing (`encryptJson`/`decryptJson` in `remoteFetch()` and `handleMessage()`), but `E2EEConnection.dek` is never populated because there is no DEK exchange protocol.
3. **No OOB key exchange** — Unlike mobvibe where the CLI prints the master secret for manual copy-paste, the current vibe-kanban has no UI or mechanism for the user to transfer the master secret to the WebUI without going through the gateway.

## Approach

**OOB Master Secret + WebUI-generated DEK (Approach A)** — inspired by mobvibe's proven pattern.

- Bridge generates master secret on first setup, prints base64 + QR code to terminal
- User manually pastes/scans into WebUI — gateway never sees any key material
- Both sides derive identical content keypair from the master secret (deterministic)
- WebUI generates a random DEK per connection, wraps it with bridge's content public key via sealed-box
- Wrapped DEK sent as first message through gateway → bridge unwraps → both sides use DEK
- All `BridgeRequest`/`BridgeResponse` payloads encrypted with DEK

## Key Hierarchy

```
master_secret (32 bytes, generated by bridge)
  ├── BLAKE2b-512(master_secret || 1 || "vkauth__") → Ed25519 auth keypair (gateway auth)
  └── BLAKE2b-512(master_secret || 2 || "vkcont__") → X25519 content keypair (DEK wrapping)
       └── DEK (32 bytes, random, per-connection, generated by WebUI)
            └── XChaCha20-Poly1305 symmetric encryption of payloads
```

## Design Changes From Current

| Aspect | Current (broken) | New |
|--------|-------------------|-----|
| Master secret origin | Generated by WebUI, sent to bridge via gateway API | Bridge generates on first setup, OOB transfer to WebUI |
| Content pk delivery | Via gateway API (MITM risk) | OOB: user copies master secret |
| Bridge encrypted path | Stub (returns early) | Full decrypt/proxy/encrypt cycle |
| DEK in connection | Always null, never set | Set after WebUI unwraps wrapped DEK |
| Gateway knowledge | Sees content public key | Zero-knowledge by construction |

## OOB Master Secret Exchange

### Bridge Side (on first setup)

When E2EE is first configured (no credentials file exists at `e2ee_config::credentials_path()`):

1. Bridge generates 32-byte master secret via `OsRng`
2. Derives auth keypair + content keypair from it
3. Saves to credentials file (path from `e2ee_config::credentials_path()`)
4. Prints to stdout with QR code (ASCII art via `qr2term` crate):
```
╔══════════════════════════════════════════════╗
║  E2EE Master Secret — paste this into WebUI  ║
╠══════════════════════════════════════════════╣
║  aB3dE5fG7hI9jK1lM3nO5pQ7rS9tU1vW3xY5zA==  ║
╚══════════════════════════════════════════════╝
[QR CODE ASCII ART]
```
5. On subsequent starts: loads existing credentials, skips generation

### WebUI Side (pairing flow)

1. User opens "E2EE Settings" dialog
2. Two input methods:
   - **Text input**: paste base64 master secret
   - **Camera scan**: scan QR code (use `jsQR` library; `BarcodeDetector` API as optional optimization where supported)
3. Calls `manager.addPairedSecret(base64Secret)` → derives content keypair, stores in localStorage
4. Success → ready to connect

### Gateway is NOT involved in this step

The gateway only handles username/password auth (Argon2id). The master secret transfer is entirely out-of-band.

## DEK Exchange Protocol

### Step 1: WebUI generates DEK and sends to bridge

```
WebUI                                          Gateway                         Bridge
  |                                              |                               |
  |  1. generate random 32-byte DEK              |                               |
  |  2. wrap DEK with content_pk (sealed-box)    |                               |
  |  3. send: { type: "forward",                 |                               |
  |            machine_id: "...",                |                               |
  |            payload: { type: "dek_exchange",  |                               |
  |                       wrapped_dek: "base64" }|                               |
  |           }                                  |                               |
  |--- forward --------------------------------->|--- forward ------------------>|
  |                                              |                               |
  |                                4. bridge unwrap DEK with content_sk          |
  |                                5. store DEK in session state                 |
  |                                6. respond: { type: "dek_ok" }               |
  |                                              |                               |
  |<-- forward({ type: "dek_ok" }) -------------|<-- forward -------------------|
  |                                              |                               |
  |  7. store DEK in connection.dek              |                               |
```

### Step 2: All subsequent messages encrypted with DEK

**WebUI → Bridge (requests):**
```json
{
  "type": "forward",
  "machine_id": "...",
  "payload": { "t": "encrypted", "c": "<base64(nonce || XChaCha20-Poly1305(BridgeRequest))>" }
}
```

**Bridge → WebUI (responses):**
```json
{
  "type": "forward",
  "payload": { "t": "encrypted", "c": "<base64(nonce || XChaCha20-Poly1305(BridgeResponse))>" }
}
```

## Bridge Encrypted Forwarding Path

### Connection session state

The DEK is stored per-gateway-connection as `Arc<Mutex<Option<[u8; 32]>>>`, following the same pattern as the existing `ws_connections` state. Each `handle_forward` call is spawned as an independent `tokio::spawn` task, so shared mutable state uses `Arc<Mutex<>>`.

```rust
// Shared alongside existing ws_connections: Arc<Mutex<WsConnections>>
let dek_state: Arc<Mutex<Option<[u8; 32]>>> = Arc::new(Mutex::new(None));
```

### Modified `handle_forward()` logic

```rust
async fn handle_forward(payload: Value, dek_state: &Arc<Mutex<Option<[u8; 32]>>>, ...) -> Result<()> {
    // Phase 1: DEK exchange (before DEK is set)
    // extract_dek_exchange checks payload.get("type") == "dek_exchange"
    // and returns payload.get("wrapped_dek") as base64 string
    if let Some(wrapped_dek_b64) = extract_dek_exchange(&payload) {
        let wrapped_dek = base64_decode(&wrapped_dek_b64)?;
        let dek = e2ee_core::unwrap_dek(&wrapped_dek, &content_sk, &content_pk)?;
        *dek_state.lock().unwrap() = Some(dek);
        send_response(tx, BridgeResponse::DekOk);
        return Ok(());
    }

    // Phase 2: Decrypt if encrypted
    let request = if e2ee_core::is_encrypted_payload(&payload) {
        let dek_guard = dek_state.lock().unwrap();
        let dek = dek_guard.as_ref().context("No DEK established")?;
        let enc_payload: e2ee_core::EncryptedPayload = serde_json::from_value(payload)?;
        let request: BridgeRequest = e2ee_core::decrypt_json(&enc_payload, dek)?;
        request
    } else {
        serde_json::from_value(payload)?
    };

    // Phase 3: Handle request (same as current)
    let response = match request { ... };

    // Phase 4: Encrypt response if DEK is set
    if let Some(dek) = dek_state.lock().unwrap().as_ref() {
        let encrypted = e2ee_core::encrypt_json(&response, dek)?;
        // Wrap EncryptedPayload in forward envelope and send via tx
        send_response(tx, BridgeResponse::Encrypted(encrypted));
    } else {
        send_response(tx, response);
    }
}
```

### Error handling

| Scenario | Behavior |
|----------|----------|
| Encrypted payload but no DEK | Return error: "No DEK established" |
| DEK exchange with invalid wrapped DEK | Return error: "Invalid wrapped DEK" |
| Decryption failure (wrong DEK / tampered) | Return error: "Decryption failed" |
| Bridge sends encrypted response, WebUI can't decrypt | WebUI logs error, drops message |

### DEK lifecycle

- **On connect**: WebUI generates DEK, sends wrapped DEK to bridge, waits for `dek_ok`
- **On disconnect**: Both sides drop DEK. WebUI sets `this.dek = null`. Bridge side: DEK state is local to `connect_and_run` scope, so it is dropped when the gateway WebSocket closes.
- **On reconnect**: New DEK exchange required. Each connection gets a fresh random DEK.

### `BridgeResponse` changes

Two new variants are needed:

```rust
pub enum BridgeResponse {
    // ... existing variants ...
    /// DEK exchange acknowledgement
    #[serde(rename = "dek_ok")]
    DekOk,
    /// Encrypted response payload
    Encrypted(e2ee_core::EncryptedPayload),
}
```

## Frontend Changes

### New `initDek()` method on `E2EEConnection`

`initDek()` registers a one-shot listener (similar to `pendingRequests`) for the `dek_ok` response, then sends the DEK exchange message. `handleResponsePayload` needs a new `case 'dek_ok'` branch that resolves the listener.

```
connectToGateway()
  → WebSocket connected, auth_ok received
  → connection.initDek(manager, machineId)
    → generate random 32-byte DEK
    → get content public key from manager (derived from paired secret)
    → wrap DEK with sealed-box(content_pk)
    → register one-shot listener for "dek_ok" response
    → send { type: "forward", payload: { type: "dek_exchange", wrapped_dek: "..." } }
    → await listener promise (timeout after 10s)
    → on "dek_ok": set this.dek = dek
    → on timeout/error: throw, connection retries
  → ready for encrypted remoteFetch()
```

`handleResponsePayload` addition:
```
case 'dek_ok':
  → resolve the one-shot DEK listener
  → break
```

### Updated `connectToGateway` in `useE2ee.ts`

```typescript
const connectToGateway = useCallback(async (...) => {
  await connection.connect({ ... });
  connection.subscribeMachine(machineId);

  // NEW: establish DEK before using the connection
  if (manager.hasPairedSecrets) {
    await connection.initDek(manager, machineId);
  }

  setConnected(true);
}, []);
```

### E2EE Pairing Dialog

Two input modes:
1. **Text paste**: textarea for base64 master secret (existing `addPairedSecret` flow)
2. **QR scan**: button to open camera, scan QR, auto-fill (uses `jsQR` library; `BarcodeDetector` API as optional optimization where supported)

## Security Properties

| Property | Status | Explanation |
|----------|--------|-------------|
| Gateway zero-knowledge | Achieved | Gateway never sees master secret, content keys, or plaintext. |
| MITM resistance | Achieved | Both sides derive content keypair from OOB-shared master secret. |
| Per-session key isolation | Achieved | Each WebUI connection gets a fresh random DEK. |
| Forward secrecy | Not achieved | If master secret leaks, all past sessions can be decrypted. |
| Post-compromise security | Not achieved | Static keys for the lifetime of the master secret. |

### Threat scenarios

| Attacker | Can they decrypt? | Why |
|----------|-------------------|-----|
| Gateway operator | No | Never sees master secret or DEK. |
| Network eavesdropper | No | All traffic over WSS + sealed-box ciphertext. |
| Attacker with gateway code access | No | No key material available to the gateway process. |
| Attacker with bridge machine access | Yes (past + future) | Has `credentials.json` with master secret; can derive all keys indefinitely. |
| Attacker who steals WebUI localStorage | Yes (past + future) | Has master secret stored in plaintext; can derive all keys indefinitely. |

## Comparison with mobvibe

| Aspect | mobvibe | This design |
|--------|---------|-------------|
| Who generates DEK | CLI (bridge side) | WebUI (browser side) |
| Who wraps DEK | CLI wraps for WebUI | WebUI wraps for bridge |
| Direction of wrapped DEK | CLI → gateway → WebUI | WebUI → gateway → bridge |
| Crypto primitives | sealed-box (X25519 + BLAKE2b + XChaCha20) | Same |
| Symmetric encryption | XSalsa20-Poly1305 | XChaCha20-Poly1305 |
| OOB method | CLI prints to terminal | Bridge prints + QR code |

The DEK direction difference is intentional — mobvibe's CLI initiates sessions, vibe-kanban's WebUI initiates. Functionally and cryptographically equivalent.
